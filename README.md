PROJECT HELIX: Institutional-Grade Infrastructure ArchitectureClassification: FinTech / Payment OrchestrationCompliance Target: PCI-DSS Level 1 Capable (Highly Isolated)Cloud Provider: AWS (Amazon Web Services)Deployment Strategy: Infrastructure as Code (Terraform) + GitOps (ArgoCD)1. THE ARENA: Network Topology (VPC Design)To achieve PCI compliance and absolute zero-trust security, the network is aggressively segmented across 3 Availability Zones (AZs) for high availability.Edge/Perimeter: AWS Route 53 $\rightarrow$ AWS WAF (Web Application Firewall for DDOS/SQLi) $\rightarrow$ Amazon API Gateway.Public Subnets: Contain ONLY NAT Gateways and Application Load Balancers (ALBs). No compute instances reside here.Private Subnets (Compute App Layer): Contains the Amazon EKS (Elastic Kubernetes Service) worker nodes hosting the Go, Node.js, and Python microservices.Isolated Subnets (Data Layer): Strictly for Databases (Aurora, DynamoDB) and Caches (Redis). No internet access whatsoever; routing happens strictly via VPC Endpoints (AWS PrivateLink).2. COMPUTE & ORCHESTRATION LAYER (Amazon EKS)We utilize a microservices architecture managed by Kubernetes, allowing independent scaling based on specific resource bottlenecks.A. The Synchronous Core (GoLang Router)Deployment: helix-sync-routerRole: Handles real-time transaction requests.Scaling Trigger: CPU Utilization & Custom Metrics (P99 Latency > 100ms).Execution: Highly replicated, stateless pods. Uses gRPC for internal microservice chatter to minimize serialization overhead.B. The Asynchronous Gateway (Node.js)Deployment: helix-async-webhookRole: Consumes gateway webhooks (Selcom, PesaPal) and handles WebSocket connections for the React Dashboard.Scaling Trigger: Incoming connection count & SQS Queue Depth.C. The Intelligence Engine (Python + LLM)Deployment: helix-ai-workerRole: Consumes from the Disputes SQS queue.Execution: Aggregates transaction data, calls AWS Bedrock (Claude 3 / Titan) for chargeback rebuttal generation, and drops the generated PDF into an S3 bucket.Scaling Trigger: SQS Queue Depth.3. STATE MANAGEMENT & DATA LAYERSeparating state is crucial to prevent database locking during high-velocity transaction spikes.A. Amazon DynamoDB (The Idempotency Vault)Purpose: Ultra-fast, single-digit millisecond lookups for transaction hashes.Execution: When the Go router receives a request, it checks DynamoDB with a ConditionExpression. If the transaction ID exists, it instantly rejects the duplicate. We use TTL (Time to Live) to automatically purge keys after 48 hours to manage costs.B. Amazon Aurora PostgreSQL (The Source of Truth)Purpose: Core relational data (Merchants, Gateway Credentials, Transaction Ledgers).Execution: Multi-AZ deployment. Optimized for ACID compliance. The Go router writes the finalized transaction state here only after the payment gateway confirms success.C. Amazon ElastiCache [Redis] (The Speed Layer)Purpose: Distributed locking, rate limiting API requests, and caching gateway configuration rules (preventing database hits for static routing rules).4. EVENT-DRIVEN MESSAGING (The Nervous System)Synchronous systems fail under load. We decouple everything non-critical to the immediate transaction.Amazon SQS (Simple Queue Service): * incoming_webhooks.fifo: Ensures strict ordering of payment state updates from Node.js to the database.dispute_generation.standard: Holds chargeback notices for the Python AI workers.Dead Letter Queues (DLQ): Mandatory for all queues. If a webhook fails to process 3 times, it is routed to a DLQ for manual engineering review.Amazon EventBridge: Used for cron jobs (e.g., triggering end-of-day reconciliation reports).5. AI & SECURITY PIPELINEAI Inference (AWS Bedrock): We DO NOT route sensitive data to public OpenAI APIs. We use AWS Bedrock so the foundation models run securely within the AWS boundary.Secrets Management (AWS Secrets Manager): API keys for Paddle, Selcom, etc., are never stored in code. They are injected into EKS Pods at runtime using the AWS Secrets and Configuration Provider (ASCP).Encryption: KMS (Key Management Service) is used to encrypt all RDS, DynamoDB, and S3 data at rest. TLS 1.3 for all data in transit.6. OBSERVABILITY & CI/CDMetrics & Tracing: Datadog or AWS X-Ray. Tracing a request from the API Gateway $\rightarrow$ Go Router $\rightarrow$ External Gateway $\rightarrow$ Node Webhook is mandatory to find latency bottlenecks.CI/CD (GitOps): * Code pushed to GitHub triggers GitHub Actions (Runs unit tests, Go linters, Python Pytest).Builds Docker images and pushes to Amazon ECR (Elastic Container Registry).ArgoCD living inside the EKS cluster detects the new image and handles a zero-downtime rolling deployment.
